経理 入金消込処理 Excel VBA

経理担当者が日々行っていた入金消込作業を、SQL Server 連携と部分和探索アルゴリズムで自動化した Excel VBA システム。

業種
卸売業 (社内経理部門)
対象部門
経理 (複数名)
実装期間
2026年1月初版、以降4ヶ月にわたり継続改善
本番稼働
2026年1月
技術スタック
Excel VBA, ADO, SQL Server, 部分和探索アルゴリズム

背景・課題

経理担当者は毎日、銀行から受信した入金明細を「どの請求に対する入金か」を突き合わせる「消込」作業を行います。

例えば「3万円の入金が来た」場合、それが「当月回収1.5万円 + 滞留1の1.5万円」の合計だったり、「予定2と予定4の合算」だったりします。これを担当者が頭の中で組み合わせを試行錯誤して判断していました。

項目は12種類 (当月回収・滞留1〜5・予定1〜6) で、組み合わせは 2の12乗 - 1 = 4,095通り。人間が全パターンを試すのは不可能で、勘と経験で「だいたいこれだろう」と判定していました。

結果として:

  • 消込作業に1日数時間
  • 判定ミス → 後工程 (請求書照合) でエラー発覚 → 戻り作業
  • 得意先名・営業担当者名を別画面で確認する手間
  • 属人化 (ベテランしか正確に消込できない)

実装した解決策

Excel をフロントエンドとして、裏側で SQL Server と連携する VBA システムを構築しました。

ボタン1: 検証 (DB → Excel)

  • 得意先CDを基準に SQL Server から得意先名・回収予定 (当月回収・滞留1〜5・予定1〜6) を取得
  • M_担当者テーブルから営業担当者名・事務担当者名を自動取得
  • 各入金行に対して 4,095通りの部分和を全探索し、入金額と一致する組み合わせを発見
  • 整合した行 → 薄黄色で塗り分け、整合しない行 → 赤塗りで目視チェック効率化

ボタン2: 取込 (Excel → DB)

  • 整合済み行を SQL Server の入金テーブルに登録
  • 伝票番号を自動採番して書き戻し
  • 確認ダイアログで対象シート名一覧を表示してから実行 (誤操作防止)

付随機能

  • 営業担当者名一括更新 (M_担当者の更新を複数シートに反映)
  • セル変更時の自動再計算
  • フィルタ適用中の安全な動作 (チェックボックス形式は復元、それ以外は警告)

効果

項目BeforeAfter削減量
消込判定 手作業で組み合わせ探索 4,095通り自動探索 (1秒) 判断時間ほぼゼロ
得意先名・担当者名の確認 別画面で都度確認 自動表示 画面切替不要
消込作業時間 数時間/日 確認のみ (30分程度) 月20〜40時間
判定ミス・戻り作業 頻発 大幅削減 後工程の品質向上
属人化 ベテランのみ正確 新人でも操作可能 業務継続性向上

継続的改善 (4ヶ月の保守履歴)

このシステムは「作って終わり」ではなく、現場のフィードバックを受けて4ヶ月にわたり改善を続けています。

  • 2026年1月: 初版リリース (コード4,169行 + 仕様書533行 + マニュアル243行を同時整備)
  • 2026年2月: 滞留・予定の表示と整合チェック / 営業・事務担当者のDB取得 / フィルタ適用時のエラー解消
  • 2026年3月: 得意先CDの重複検出ロジック改善
  • 2026年4月: 取込確認ダイアログに対象シート名一覧を表示 (UI 改善)

各コミットで必ず VBA仕様書.md と操作マニュアル.md を同時更新する習慣を維持。属人化を最小化しています。

得られた学び

  • 「枯れた技術」が一番強い場合がある — Excel VBA は時代遅れと見られがちだが、現場が日々触っているツールで完結することで、教育コスト・移行コストがゼロ
  • アルゴリズムは小さくても効く — 4,095通りの部分和探索は学部1年レベルだが、現場では「考えなくていい状態」を作る決定打になる
  • ドキュメント並走更新の習慣 — コード追加と同時に仕様書・マニュアルも更新するルールを徹底することで、3年後の自分や引き継ぎ担当者を救う
  • UI への気配り — 「対象シート名一覧をダイアログに表示する」だけの修正でも、誤操作リスクを大幅に減らせる
