異常受注検知システム

受注データから異常値を自動検知し、業務担当者がブラウザで確認・対応できるシステム。ゼロから1週間で初期版をリリースし、その後5ヶ月にわたり改良を継続中。

業種
卸売業 (社内システム)
対象部門
営業部・営業事務部
実装期間
初期版: 1週間 / 改良継続: 5ヶ月以上
本番稼働
2026年1月 (初版) / 2026年4月 (Linux 移行版)
技術スタック
Python, Flask, Vue.js (Alpine.js), SQLite, Docker, Prefect 3.x, 統計分析

背景・課題

受注データの中には、まれに「通常の取引量と大きくかけ離れた発注」が含まれます。例えば:

  • 通常 月1ケースの取引先から、突然 50ケースの注文
  • 新規取引なのに、いきなり大口発注
  • 担当者の入力ミスによる桁違いの数量

これらを放置すると、誤発注による在庫過多、機会損失、取引先トラブルにつながります。一方、すべての受注を人の目で確認するのは非現実的でした。

そこで「自動的に異常値を検知して、担当者にアラートを出す」仕組みを構築することにしました。

実装した解決策

1. 統計分析による異常検知

  • 過去の取引データから、得意先×商品ペアごとの「通常範囲」を統計的に算出
  • 直近平均と比較して、3倍以上の乖離があれば「異常」として検知
  • 新規取引 (履歴ゼロのペア) は「袋入数 × N」を基準値に設定 (Phase 1)、将来は加重中央値方式へ移行予定 (Phase 2)
  • 基準日 (baseline_cutoff) を業務イベント (石油供給ひっ迫前) で区切り、特殊期を除外

2. ブラウザ UI (カード型)

  • Flask + Vue.js (Alpine.js) で軽量な SPA を構築
  • 検知された受注を「カード」として表示、業務担当者がクリックして「確認済み」「対応済み」を記録
  • 確認済みカードは絶対に再修正しない設計 (業務データの一貫性を守る)

3. データ永続化の段階移行

  • 初版: Markdown ファイルベース (実装スピード優先)
  • Phase 1: SQLite 移行 (整合性・検索性向上)
  • 段階移行時、新旧両方で並行計算し、結果が完全一致するまで切り替えない

4. 運用安定化 (watchdog 二段構え)

  • Plan A: 旧 PC 側で stale 15分検知 → 通知
  • Plan B: 新サーバー側で例外発生 → 通知
  • どちらか一方が動かなくなっても、もう片方で検知できる二重防御
  • ゾンビ run 自動 Cancel の Prefect Automation も併設

5. 89 ↔ 235 のサーバー移行

  • Windows PC ローカル運用 → Ubuntu Linux サーバー (Docker Compose) へ移行
  • 二重稼働期間を設けて、結果が完全一致することを確認してから Cutover
  • 移行後の 89 側のクリーンアップ (スタートアップ vbs のコメントアウト、タスクスケジューラ削除) まで責任を持って実施

効果

項目BeforeAfter
異常受注の発見 後工程で発覚 (出荷後・請求後) 受注直後に自動検知
機会損失リスク 大ロット誤受注で在庫圧迫 事前確認で誤受注を防止
営業担当者の負担 全受注を目視確認 異常分のみ確認 (大幅削減)
新規取引の検知 埋もれて気づかない 専用カードで可視化
運用安定性 (該当なし、新規構築) watchdog 二段構えで障害検知

得られた学び

  • 1週間でリリースし、5ヶ月かけて改善 — 初版は完璧でなくていい。早くリリースして現場の反応を見ながら改良するほうが結果的に良いものになる
  • 業務イベントを基準値に組み込む — 「石油供給ひっ迫前」のような業務側の出来事を統計の基準日にすることで、純粋な統計値より業務実態に合った検知になる
  • 段階移行時の並行検証 — 新旧両方を動かして結果が完全一致するまで切り替えないルールにすることで、移行時の事故をゼロにできる
  • watchdog の二段構え — 「監視する仕組み自体が止まる」リスクを想定し、独立した2系統で監視することで信頼性を確保
  • 確認済みデータは触らない — 業務データの一貫性を守るため、「自動処理が確認済みカードを再修正しない」原則を厳守
