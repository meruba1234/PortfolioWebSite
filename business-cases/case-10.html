業務自動化基盤の構築 (Linux + Docker + Prefect)

個別ジョブを動かすだけのツール群ではなく、社内業務全体を支える「司令塔」としての自動化基盤を、Ubuntu Linux + Docker Compose + Prefect 3.x で構築。

業種
卸売業 (社内インフラ)
対象
全部門の業務システムを連携
実装期間
計画〜本番稼働まで 約3ヶ月
本番稼働
2026年4月 (89↔235 二重稼働 → 順次 Cutover)
技術スタック
Ubuntu Linux, Docker Compose, Prefect 3.x, nginx, Python 3.x, SSH 鍵認証

背景・課題

当初、業務自動化ジョブはすべて Windows ローカル PC (89) のタスクスケジューラで動いていました。

これには深刻な問題がありました:

  • 単一障害点: 89 PC が止まれば全業務が止まる
  • 連鎖故障: FAX 複合機エラー → wscript 溜まり → 異常検知も失敗 → PC 再起動が必要
  • 監視不能: ジョブの状態を集中的に見られない
  • 属人化: 設定がローカル PC に分散
  • 拡張性なし: 新規ジョブを追加すると PC が重くなる

これらを解決するため、独立した Linux サーバー上に集中型の自動化基盤を構築することにしました。

実装した解決策

1. 235 サーバー構築 (Ubuntu + Docker Compose)

  • Ubuntu Linux サーバー (192.168.161.235) を新規構築
  • Docker Compose で Prefect Server + Prefect Worker + Flask Server + nginx を一括管理
  • batch-files (共通ライブラリリポジトリ) を automation-worker / fax-viewer に bind mount
  • 共有フォルダ (UNC) は smbprotocol で透過アクセス

2. Prefect 3.x による集中管理

  • 全ジョブを Prefect Flow として登録
  • ジョブ間の依存関係・スケジュール・リトライを一元管理
  • Web UI で実行状況・ログ・エラーを集中確認
  • Secret Block / Variable で認証情報を安全管理 (.env 廃止計画)

3. 監視・障害対応 (watchdog 二段構え)

  • Plan A: 旧 89 PC 側で stale 15分検知 (ローカルウォッチドッグ)
  • Plan B: 235 サーバー側で except 通知 (リモートウォッチドッグ)
  • どちらかが死んでも、もう一方で検知
  • ゾンビ run 自動 Cancel の Prefect Automation 2件 (異常検知12分/全flow20分)

4. nginx リバースプロキシ

  • 各サービス (Prefect UI / Flask / FAX viewer / 異常検知 UI / 商品不足管理) を 1 つのドメインに集約
  • BOM なし UTF-8 で設定ファイル管理 (PowerShell の Out-File 罠を回避)

5. 89 → 235 段階移行 (Cutover)

  • 二重稼働期間: 89 と 235 両方で動かして結果を比較
  • SQLite データは完全一致を確認してから切替
  • Cutover 後、89 のタスクスケジューラを全削除 (2026/5/13)
  • スタートアップ vbs も Flask 5001/5002 をコメントアウト

6. SSH 鍵認証によるデプロイフロー

  • 89 PC git bash から 235 へ SCP / SSH で安全にデプロイ
  • SSH 鍵の保管場所と PC 移行手順を文書化 (将来の引き継ぎ対応)

7. 設計方針ドキュメントの整備

  • 命名規則 (フォルダ英語 snake_case、JOB_NAME / Deployment 名は日本語)
  • ログ実装指針 (LOG-GUIDE-FINAL-V1 / schema v1.0.0)
  • クラス設計方針 (1クラス1役割)
  • Prefectスキル使い分けガイド
  • ファイルIO方針

得られた学び

  • 「ツール集合」ではなく「基盤」を作る — 個別ジョブを足し算するだけでなく、共通ライブラリ・命名規則・設計方針を整えることで、19本のジョブが管理可能な規模になる
  • watchdog 二段構えの威力 — 監視する仕組み自体が止まる可能性を想定し、独立した2系統で監視する設計が運用安定化の決定打
  • 二重稼働期間の必要性 — Cutover を一発で行うのは危険。両方動かして結果を比較してから切替えるルールを徹底
  • 引き継ぎを前提とした設計 — 自分が辞めても次の人が続けられるよう、SSH 鍵保管場所・サーバー構成・デプロイ手順を文書化
  • ゾンビ run 自動 Cancel — Prefect Automation で 12分・20分の閾値を設定。サーバー側で自動 Cancel することで、後続ジョブのブロックを防ぐ

備考: この基盤の上に乗っているジョブ

本基盤の上で、以下のような業務改善ジョブが稼働しています:

  • 取引先A 追加返品処理 (Playwright 画面入力代行)
  • 取引先B EDI発注処理 (販管DB連携)
  • EOS受注 マルチサプライヤー 4社
  • BIツール × 出荷目安レポート
  • 月次請求金額内訳レポート
  • FAX 履歴ビュアー
  • 異常受注検知システム
  • 商品不足管理サイト
  • セキュリティチェック
  • ... 他多数

つまり本基盤は、上記すべての業務改善案件を支える「司令塔」として機能しています。
