商品不足管理サイト

欠品予兆のある商品を Web で一覧化し、商品部・営業部が状況をリアルタイムに共有・対応できるシステム。差分10分毎 + 全件毎日6:05の自動更新で常に最新状態。

業種
卸売業 (社内 Web ツール)
対象部門
商品部・営業部
実装期間
初版〜本番稼働 約2〜3週間
本番稼働
2026年5〜6月 (商品部レビュー中)
技術スタック
Python (Flask), Vue.js (Alpine.js), SQLite, SQL Server (販管DB), Prefect 3.x, nginx

背景・課題

卸売業では、在庫不足が発生すると以下のような問題が起こります:

  • 取引先への納品遅延 → 信頼失墜
  • 営業の機会損失
  • 仕入れタイミング遅れによる調達コスト増

従来は、商品部が販管システムを叩いて個別に確認していましたが:

  • 確認する人によって判断基準がバラバラ
  • 営業部と情報共有のタイムラグ
  • 「見落とし」による在庫切れ事故

これを「全社で同じ情報をリアルタイムに見る」仕組みが必要でした。

実装した解決策

1. 差分検出 + 全件再構築のハイブリッド更新

  • 差分 10分毎: 直近変動を検出して即時反映 (Prefect Flow)
  • 全件再構築 毎日 6:05: 整合性を保つために日次でフルスキャン
  • 差分処理だけだと過去データの逆転を見逃すため、両方を組み合わせ

2. SQLite による軽量永続化

  • 販管 DB (SQL Server) は重い & 共有資源
  • SQLite に集計結果を永続化することで、Web 表示は SQLite を叩くだけで完結
  • SQLite ファイル本体・バックアップはストレージ整理の対象外として保護

3. Web UI (Vue.js + Alpine.js)

  • 商品ごとの状態をカード型 UI で表示
  • 担当者・状態 (注文中・確認中・欠品確定など) でフィルタリング
  • 商品部が状態更新 → 営業部も同じ画面で見られる

4. 配信基盤

  • 235 サーバー上で nginx + Flask
  • URL は /shortage/ で社内 8082 ポートから提供
  • 運用ジョブ (diff / full) は Prefect Deployment 登録済み

5. 異常検知システムへの波及

  • 本基盤の「在庫・滞留在庫」情報を、異常受注検知システム側でも表示する計画
  • 滞留フラグは販管側にないため、shortage_master_sync 基盤に乗せる設計を選択

効果

項目BeforeAfter
不足商品の確認 各自が販管を叩いて手作業 Web 1 画面で全件一覧
商品部と営業部の情報共有 口頭・メール・タイムラグあり 同じ画面でリアルタイム共有
欠品見落としリスク 頻発 大幅削減 (10分毎更新)
判断基準の統一 担当者依存 システムで一元化
営業の機会損失 気づいた時には在庫切れ 早期把握 → 早期対応

得られた学び

  • 差分処理だけでは不完全 — 差分検出はリアルタイム性が高いが、過去データの遡及変更を見逃す。日次フルスキャンとの組み合わせで整合性を保証
  • 重い DB と軽い表示用 DB の分離 — 販管 SQL Server (重い) を Web 表示で叩くのは危険。SQLite に集計済みデータを置くことで Web 応答速度と DB 負荷の両立
  • カード型 UI の柔軟性 — 状態が多次元 (商品 × 担当者 × ステータス) なデータは、テーブルよりカード型のほうが業務担当者に好まれる
  • 業務側のレビューを挟む — 本番稼働しても「商品部レビュー」フェーズを経て、業務側の声を反映してから完成扱いにする慎重さが信頼を生む
