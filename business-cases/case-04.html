取引先B EDI発注処理 自動化

大手外食チェーン取引先から契約3社経由で送られてくる EDI 発注 CSV (単価・金額未入力) を読み取り、自社販売管理 DB から単価を決定して金額計算し、CSV を返送するジョブ。

業種
外食チェーン (取引先B、契約3社経由)
対象部門
営業事務 (旧手動運用)
実装期間
約2週間
本番稼働
2026年6月 (毎週火曜・土曜 10:05前後)
技術スタック
Python, Prefect 3.x, SQL Server (pyodbc), Linux, .bat 連携

背景・課題

取引先B (大手外食チェーン) は、契約3社の中間業者経由で「単価・金額が空欄の発注 CSV」を週2回 (火・土) 送ってきます。受け手は:

  1. CSV を受領
  2. 各明細について、社内の販売管理 DB から「本体売価 × EOS 換算数量」を計算して単価を決定
  3. 単価 × 数量 で金額を計算
  4. CSV に書き戻して返送

これを手動でやると、明細数百行 × 週2回 = 営業事務さんの作業時間として無視できないボリュームでした。 さらに、計算ロジック (本体売価 × EOS換算) が複雑で、ルール変更の度に営業事務さんを教育する負担もありました。

実装した解決策

1. 既存システムのロジック移植

販売管理システム (VB.NET) 内の「csv_to_db_v3.py / YNJTM32」の EOS 振り分け正規ルールを解読し、Python で完全互換実装。

  • 本体売価 × EOS 換算数量 (税区分問わず)
  • D_売上_D 登録用の内税数量計算 (本ジョブでは数量列を書き換えないため不要、と検証で確認)
  • 書き換えるのは「単価 (列26)」と「金額 (列27)」の2列のみ。数量列はノータッチ

2. 起動経路の最適化

  • 当初は Cron 10分毎ポーリングで設計
  • 実運用で EOS 名人 (社内 EDI 取込ツール) からの .bat 起動経由に一本化
  • Prefect API を .bat から呼ぶことで、社内オペレーターは EOS 名人を使い続けられる

3. データ整合性と運用安定化

  • 厳格モード (1件でも単価未決定があれば全体エラー)
  • 括弧付きファイル ((1)(2)) の自動リネーム + ChatLuck WARNING 通知
  • 失敗 CSV は隔離フォルダに自動退避、inbox/order 分離設計
  • リトライ無効化 (retries=0) — 中途半端な再試行で Completed に化けるバグを回避

4. 性能最適化

  • prefetch 化により 40.58 秒 → 5.59 秒 (7.3倍高速化)
  • 本番 DB で結果完全一致を検証してから切り替え

5. 認証情報の安全管理

  • 共有 .env から Prefect Variable / Secret に統一
  • 実行環境表示も Variable 化

得られた学び

  • 既存システムのソースコードを必ず読む — 「だいたい同じ計算」では事故る。csv_to_db_v3.py / YNJTM32 と突き合わせて検証することで、誤認していたルールを修正できた
  • 書き換える列を仕様書に明示 — 「数量列はノータッチ、単価・金額のみ書き換え」を仕様書冒頭に明記。後から見直すときに混乱を防ぐ
  • 運用側の慣習を尊重 — EOS 名人 (既存 EDI ツール) を残しつつ、その中から Prefect を呼ぶ設計にしたことで、オペレーターは何も変えずに済む
  • パフォーマンス改善は実測で判断 — 7.3倍の高速化を「速くなった気がする」ではなく、本番 DB での結果完全一致 + ベンチマークで検証
