EOS受注 マルチサプライヤー自動化 (4社一括)

業態の違う 4 サプライヤーサイトから EOS 受注 CSV を自動取得 + 出荷確定 CSV を自動アップロード。各サイトごとに異なる Web UI を Playwright で個別に攻略。

業種
外食・小売・水産など複数業態
対象部門
営業事務
実装期間
各サイトあたり 1〜2週間 × 4社
本番稼働
2026年4月〜6月 (順次)
技術スタック
Python, Playwright, smbprotocol (Linux→SMB共有), Prefect 3.x, ChatLuck通知

背景・課題

大手取引先各社は EOS (電子発注システム) を採用しており、毎週決まった曜日に受注確認・出荷確定のための Web 画面操作が必要でした。

  • 取引先 D (水産): 月水金 09:30 にサプライヤーサイトから受注 CSV ダウンロード
  • 取引先 E (グループA): 月金 11:00 に EOS データを取り込み、DB の M_EOS* マスタに登録
  • 取引先 F + 取引先 G (グループ会社、ベーカリー系): 火曜 09:00 にグループ一括配信の受注 CSV をダウンロード、火曜 11:30 に出荷確定 CSV アップロード

すべて手動運用だと、担当者が時間に縛られ、休めない・ミスると取引先トラブルというストレスフルな状況でした。

実装した解決策

1. 各サイトを個別に Playwright で攻略

各サプライヤーサイトは UI も認証もまったく異なるため、それぞれ個別に Playwright スクリプトを設計。

  • 取引先 D (水産) — シンプルなログイン後、月水金 09:30 のスケジュール実行
  • 取引先 E (グループA) — 既存ツール (MasterMainte.exe) の代替として、DB の M_EOS業者変換 / M_EOS商品変換 / M_EOS数量換算 に自動追加登録
  • 取引先 F + G — グループ一括配信のため 1ジョブで両社分一度に取得 (受注 + 出荷確定の2 Flow に分離)

2. 共通の検出ロジック

  • Setcomb (検索条件自動入力) 完了を expect_response で待機
  • 0件判定を CSV 出力ボタンの enabled 状態で判定 (最大20秒ポーリング)
  • 重複検知時は order/CSV を smbclient で自動削除 + Prefect リトライ無効化

3. Linux サーバーからの UNC アクセス

  • smbprotocol で UNC 共有フォルダへ直接アクセス
  • UNC パスは Prefect Variable で管理 (forward slash 形式)

4. 出荷確定処理 (アップロード系)

  • codegen 自動生成だった dialog.dismiss() が Cancel 扱いになる罠を発見・修正
  • 確認 dialog は accept(OK) に変更
  • 取込実行後の待機を 8秒に延長して安定化

5. 通知設計

  • 成功・失敗・重複検知をすべて ChatLuck に整形通知
  • 失敗時の WARNING は件数のみ通知 (詳細は Prefect UI で確認)

効果

項目BeforeAfter
担当者のサイト操作時間 4社 × 15〜30分 × 週2〜3回 ゼロ (確認のみ)
月間削減時間 — 約15時間/月
時間制約 決まった時間に必ず PC 前にいる必要 スケジュール実行で解放
担当者休暇時の業務継続 引き継ぎ必須・ミスリスク 自動継続
取引先トラブルリスク 処理漏れ・遅延あり スケジュール厳守で大幅削減

得られた学び

  • Web 自動化は UI 仕様の罠との闘い — Playwright codegen で自動生成したコードが正解とは限らない。dialog.dismiss が Cancel 扱いになるなど、運用してから初めてわかる罠がある
  • サイトごとに最適な待機戦略 — sleep 固定、expect_response、ボタン enabled ポーリングなど、サイト特性に応じて使い分ける
  • Linux からの UNC アクセスは smbprotocol が決定打 — smbclient 直叩きより安定する
  • EOS 名人 (既存ツール) を残しつつ Prefect で前処理 — 既存運用を尊重した上で、その前段だけ自動化することで業務側の混乱を最小化
