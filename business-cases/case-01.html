取引先A 追加返品処理 自動化

営業事務が毎朝1〜2時間費やしていた手作業を、Web スクレイピングと画面入力代行で完全自動化したケース。

業種
水産卸 (取引先A)
対象部門
営業事務 (1名 + バックアップ1名)
実装期間
約2.5週間
本番稼働
2026年6月
技術スタック
Python, Playwright, Prefect 3.x, Dr.Sum, kintone API, ChatLuck (社内チャット)

背景・課題

取引先A様向けの「追加返品処理」は、店頭在庫の調整に伴う追加発注・返品データを毎朝集約して、社内システム (kintone) に登録する業務です。

従来は以下の流れで、営業事務が毎朝1〜2時間費やしていました:

  1. BI ツール (Dr.Sum) にログインして、納品日条件で Excel をダウンロード
  2. Excel の明細を見て kintone「発注・調整処理」アプリへ手入力 (1件あたり数分)
  3. レコードごとに「追加返品確定【一括変更】」ボタンを押下して、別アプリに展開
  4. 処理済み Excel を共有フォルダに移動

件数が多い日 (30〜80件) は2時間以上かかり、入力ミス・確定漏れのリスクもありました。朝の繁忙時間と重なるため、心理的負担も大きい状態でした。

実装した解決策

Prefect 3.x で日次ジョブを構築し、以下の処理を完全自動化しました。

  • BI ツールからの自動ダウンロード — Playwright で BI ツールにログインし、納品日条件を自動設定して CSV を取得
  • kintone への画面入力代行 — kintone API ではなく、あえて画面入力を Playwright で代行 (業務側の検算ロジックが画面側にあるため)
  • 金額検証 + 諸口振替フォールバック — 金額不一致時は自動的に諸口扱いに振替
  • 結果通知 — 成功・失敗・スキップを社内チャット (ChatLuck) に整形して通知
  • 当日以外スキップ機能 — Phase 2 の確定処理は当日分のみ対象にして、過去データの誤確定を防ぐ

当初は kintone API 経由で実装していましたが、業務側の検算ロジック (店舗コード正規化、振替コード判定など) が画面側に埋め込まれていたため、後に Playwright での画面入力代行に一本化しました。

業務側からの声

「朝のバタバタが無くなって本当に助かっています。もうこの自動化なしでは仕事できないです。」
— 営業事務担当 (本番稼働1ヶ月後)

得られた学び

  • API より画面入力代行が正解の場面がある — 業務側の検算ロジックが画面側に埋め込まれていると、API ではバイパスしてしまう。Playwright で実際にユーザー操作と同じ経路を通すほうが安全
  • 金額検証 + フォールバック設計の重要性 — 自動化は「絶対に正しい」が前提だと脆い。検証 NG 時のリカバリ経路を設計時から組み込むべき
  • 当日ガード機能 — 過去データの再処理事故を防ぐため、処理対象日のホワイトリスト化が有効
