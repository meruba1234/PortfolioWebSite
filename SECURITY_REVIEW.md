# セキュリティレビュー（PortfolioWebSite）

## 対象
- 静的HTML/CSSと、リポジトリ内で配布されているAPKファイル。

## 指摘事項

### 1. 直接メールアドレスの公開によるスパム・収集リスク
- `mailto:`リンクで個人メールアドレスが平文で公開されています。
- ボットに収集され、スパム送信やフィッシングの標的になりやすくなります。

**該当箇所**
- `contact.html` の `mailto:meruba1234@gmail.com`。

**推奨対応**
- フォーム＋サーバー/フォームサービス経由に変更し、メールアドレスの直接露出を避ける。
- どうしても公開する場合は画像化や難読化を検討（根本対策としてはフォーム化が望ましい）。

### 2. APKをリポジトリ直下で配布していることによるサプライチェーンリスク
- `QuizVoyage.apk` / `app-release.apk` / `QuizVoyageMath.apk` がリポジトリに配置されています。
- 改ざん検知情報（ハッシュ値・署名情報・配布元の検証手順）が明示されていないため、利用者が真正性を確認しづらい状態です。

**該当箇所**
- `android-app.html` でAPKダウンロードを案内。
- ルート配下にAPKファイルが存在。

**推奨対応**
- GitHub Releases などの配布チャネルに移し、SHA-256ハッシュと署名検証手順を公開する。
- 可能ならGoogle Play等の信頼できる配布経路を優先する。

### 3. クライアント側での防御方針（CSP等）がドキュメント上で未確認
- HTMLにCSP（`Content-Security-Policy`）やその他セキュリティヘッダ設定の記述はありません。
- 静的サイトはサーバー設定側で制御するのが一般的ですが、現状のリポジトリからは設定有無が確認できません。

**該当箇所**
- `portfolio.html` / `itskills.html` / `contact.html` / `android-app.html` の `<head>`。

**推奨対応**
- ホスティング設定で以下ヘッダを有効化: `Content-Security-Policy`, `X-Content-Type-Options`, `Referrer-Policy`, `X-Frame-Options`（または `frame-ancestors`）。
- 必要に応じて `Permissions-Policy` も設定する。

## 総評
- 動的な入力処理がほぼないため、典型的なSQLiやサーバーサイドRCEのリスクは低い構成です。
- 一方で、**公開情報管理（メール露出）**と**配布物の真正性担保（APK）**は優先的に改善すると安全性が大きく向上します。
